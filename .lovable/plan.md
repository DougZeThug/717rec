
## Goal

Expose 717rec as a remote MCP server at `https://wcitdamvochthvxvtxyb.supabase.co/functions/v1/mcp` so Claude Code / Codex can connect ("Add MCP server" → paste URL → sign in with your 717rec account). Every call is authenticated as a real 717rec user; admin-only tools stay admin-only via the existing `has_role()` check.

## Approach

Use `@lovable.dev/mcp-js` — the Supabase Edge Function is generated by a Vite plugin from three authored pieces:

1. `src/lib/mcp/tools/*.ts` — one file per tool
2. `src/lib/mcp/index.ts` — the MCP definition (`defineMcp`) with Supabase OAuth 2.1 as the auth verifier
3. `mcpPlugin()` line added to `vite.config.ts`

Tokens are verified by the SDK against Supabase's OAuth issuer; each tool forwards the caller's token to Supabase so RLS runs as that user. No `service_role` key is used anywhere in the MCP code.

## Consent route

Supabase's authorization flow redirects the connecting client (Claude, Codex) into 717rec at `/.lovable/oauth/consent?authorization_id=...`. I'll add:

- `src/pages/OAuthConsent.tsx` — a small page that loads the authorization details from `supabase.auth.oauth.getAuthorizationDetails`, shows an "Approve / Deny" screen for the client name, and forwards the resulting redirect URL.
- A route for `/.lovable/oauth/consent` in `src/App.tsx`.
- A small tweak to `src/pages/Auth.tsx` so an unauthenticated visitor to the consent URL is bounced to `/auth?next=<full consent url>` and returned there after sign-in (password sign-in, signup `emailRedirectTo`, AND Google `redirect_uri` all need to honor `next` — otherwise the OAuth handshake bounces to `/` and Claude/Codex silently fails). `next` will be validated as a same-origin relative path.

## Tools shipped in v1

Read-only league data (RLS = anon-visible tables, but the whole server still requires sign-in):

- `list_teams` — active-season teams, optionally filtered by division
- `get_team` — team detail by slug or name (record, division, power score)
- `get_standings` — current standings, optional division filter
- `get_schedule` — upcoming or recent matches, optional team filter
- `get_playoff_brackets` — active-season brackets + status
- `head_to_head` — H2H record for two teams (wraps existing RPC)

My team / my matches (uses `ctx.getUserId()` + user-scoped Supabase client):

- `get_my_team` — team the signed-in user is a member of this season
- `get_my_upcoming_matches`
- `get_my_recent_matches`

Admin tools (each handler checks `has_role(user_id, 'admin')` first, returns `isError` otherwise):

- `get_counter_drift` — wraps `v_counter_drift`
- `reconcile_counter_drift` — wraps the `reconcile_team_counters` RPC (destructiveHint=false, but confirmed via the RPC's own admin gate)
- `get_ops_health` — wraps `OpsHealthService`
- `finalize_bracket_standings(bracket_id)` — wraps existing RPC

Each tool gets a clear `title`, one-sentence `description`, and correct `annotations` (`readOnlyHint`, `destructiveHint`) — that's what Claude/Codex show in their tool picker.

## Supabase OAuth 2.1 activation

Call `supabase--configure_oauth_server` to enable the OAuth 2.1 authorization server + dynamic client registration on this project's Supabase. If it fails because this Supabase project is external/bring-your-own and doesn't have OAuth 2.1 enabled, I'll stop and tell you what to enable in your Supabase dashboard — I will NOT route around it by widening RLS, using the service-role key, or making tools public.
